# Chroniq Iteration Plan

- Date: 2026-03-12
- Topic: 单条多行输入 + 单次输入自动拆分多条记录
- Status: Proposed

## Background

当前 Chroniq 的输入模型偏向两类：

1. 单条内容直接 `chroniq add "..."` 写入
2. 无内容时进入“多行 = 多条记录”的批量录入模式

这和当前需求之间存在一个语义缺口：

1. 用户可能想保存“一条记录，但正文内部带换行”
2. 用户也可能想“一次输入一整段内容，再由系统帮忙拆成多条记录”

这两个需求不能继续共用同一个“无参多行输入”语义，否则行为会冲突。

---

## Product Decision

采用“显式输入模式分离”方案，而不是继续让 `chroniq add` 的无参行为承担多种含义。

### 结论

1. 保留当前批量模式：`chroniq add` 无参数时，仍表示“多行输入，多条记录”
2. 新增单条多行模式：明确表示“一个输入块，保存为一条记录”
3. 新增单次输入自动拆分模式：明确表示“一个输入块，预处理后落成多条记录”

### 为什么这样做

- 不破坏现有用户习惯
- 不让“多行”同时代表两种相反语义
- 仍然符合 Chroniq 的产品边界：增强输入层，而不是做复杂知识整理系统
- append-only 约束不受影响

---

## Solution

## 需求 1：单条内容支持换行展示

### 方案

新增单条多行输入模式，例如：

```bash
chroniq add --multiline
chroniq add --stdin
chroniq add --file note.md
```

行为约定：

- `--multiline`：打开编辑器，整段内容保存为一条 entry
- `--stdin`：把整个标准输入视为一条 entry，而不是按行拆开
- `--file`：把文件全文作为一条 entry 写入

存储层：

- 继续使用现有 `content: string`
- 不改 append-only
- 允许 `content` 原样包含 `\n`

展示层：

- `today` / `list` 默认继续展示单行摘要，避免刷屏
- 新增 `--full` 或 `--raw` 视图时，按原换行渲染正文
- JSON 输出不做特殊处理，直接保留真实换行

### 设计原则

- 输入时保真
- 展示时分为“摘要视图”和“全文视图”
- 默认输出仍然适合 CLI 快速浏览

---

## 需求 2：一次输入单条内容，但按语义拆成多条记录

### 方案

新增“单输入块 -> 多记录”的拆分模式，例如：

```bash
chroniq add --split auto
chroniq add --split bullets
chroniq add --split paragraphs
```

推荐按两层能力实现：

### 第一层：确定性拆分（优先实现）

基于结构而非模型做拆分，规则包括：

- 空行分段
- Markdown / 文本 bullets
- 编号列表
- checkbox 列表
- 明显的时间日志前缀

示例：

```text
- 跟张三同步定价
- 修复 add 命令边界情况
- 想到一个新的 capture flow
```

可以稳定拆成 3 条记录。

### 第二层：拆分预览（强烈建议）

在真正写入前展示：

- 即将生成多少条
- 每条预览内容
- 是否确认落盘

这样可以避免误拆。

### 第三层：可选语义拆分扩展（后置）

如果后面确实需要更强的“语义理解后拆分”，不要直接绑死云模型，而是做成可选适配层：

- 默认关闭
- 明确是 enhancement，不是基础能力
- 只在确定性拆分不够用时再加

这一步优先级低，因为它更容易让产品偏向“智能整理系统”。

---

## Priority Tasks

### P0

1. 明确输入模式契约
   - `chroniq add` 无参数继续表示“批量多条”
   - `chroniq add --multiline` 表示“单条多行”
   - `chroniq add --split <mode>` 表示“单输入块拆多条”

2. 实现单条多行输入能力
   - 支持 editor / stdin / file 三种入口中的最小闭环
   - 推荐先做 `--multiline` + `--stdin`

3. 实现展示层全文视图
   - `today --full`
   - `list --date YYYY-MM-DD --full`
   - 保持默认摘要视图不变

### P1

4. 实现确定性拆分器
   - `auto`
   - `bullets`
   - `paragraphs`

5. 为拆分模式增加预览确认
   - 输出条数
   - 输出每条摘要
   - 用户确认后再写入

6. 补输入模式测试
   - 单条多行写入
   - 全文视图渲染
   - 自动拆分结果稳定性

### P2

7. 评估是否需要 schema 扩展
   - 例如 `input_mode` / `content_format`
   - 仅在后续查询或导出确实需要时再加

8. 评估可选语义拆分扩展
   - 仅作为显式 opt-in
   - 不作为默认主路径

---

## Recommended Execution Order

1. 定义命令契约和 help 文案
2. 落地 `--multiline`
3. 落地 `--stdin`
4. 落地 `today/list --full`
5. 落地确定性拆分器
6. 落地拆分预览确认
7. 补测试和文档

---

## Acceptance Criteria

### 对需求 1

- 用户能把带换行的正文保存为单条记录
- 默认列表视图仍然简洁
- 显式全文视图能按原换行展示
- 导出 JSON 时换行不丢失

### 对需求 2

- 用户能一次输入一整块文本
- 系统能基于结构稳定拆成多条记录
- 写入前能看到预览并确认
- 误拆风险可控，且不会影响 append-only

---

## Risks

1. `add` 语义混乱
   - 如果不做显式模式区分，用户会分不清“多行是一条还是多条”

2. 默认输出刷屏
   - 如果默认直接渲染多行正文，`today` / `list` 可读性会明显下降

3. “语义拆分”被过度承诺
   - 真正的语义理解如果过早引入模型，会偏离本地优先和轻量定位

4. 批量模式与单条多行模式相互污染
   - 必须在命令层把两者拆开，不要靠猜测输入内容决定行为

---

## Final Recommendation

这两个需求都应该做，但实现顺序必须是：

1. 先解决“单条多行输入与展示”
2. 再解决“单输入块自动拆分多条”

原因：

- 需求 1 是基础保真能力，风险低、收益直接
- 需求 2 本质是解释和拆分，复杂度更高，必须建立在清晰输入契约之上

一句话方案：

> 用显式输入模式把“单条多行”和“单次输入拆多条”分开实现，先保真，再做拆分。
